# Echo 深色玻璃聊天工作台设计

日期：2026-03-23  
状态：已确认，待设计评审与实现规划

## 1. 背景与结论

本次设计基于用户提供的 Apple Music 风格参考图推进，但目标不是将 Echo 改造成媒体内容首页，而是借用其桌面级深色玻璃质感、左侧原生导航结构与安静的层次控制，重塑 Echo 当前聊天工作台的视觉外壳。

最终方向结论如下：

- Echo 继续保持以聊天为主的 AI 工作台语义。
- 整体应用壳层切换为深色玻璃桌面体系。
- 左侧栏采用“上方固定导航 + 下方会话列表”结构。
- 顶部栏保持为轻工具栏，只承担少量关键操作。
- 消息区与输入区进入新的深色体系，但不推翻现有聊天交互模型。

本次改版的重点是“借风格，不换产品类型”。

## 2. 设计目标

- 让 Echo 从当前偏纸面、编辑感的 Atelier 轻暖风格，转向更接近原生桌面应用的深色沉浸式工作台。
- 借助磨砂半透明、深色层叠和克制的高光，建立更稳定的 macOS 级产品质感。
- 保留聊天工作流作为界面主轴，避免被“内容首页”或“媒体库”结构带偏。
- 在不破坏已有业务逻辑与组件边界的前提下，为后续 UI 重绘提供清晰边界。

非目标：

- 不做音乐播放器式首页。
- 不引入推荐流、大卡片内容 feed 或媒体浏览式信息架构。
- 不为了追求拟态而新增复杂控制条、播放栏或与 AI 工作流无关的装饰模块。
- 不重构会话、流式消息、附件、Skills、MCP 等核心逻辑。

## 3. 产品定位与体验原则

本次设计中的 Echo 仍然是一个用于发起对话、延续会话、整理 AI 输出的工作台，而不是内容消费型产品。

体验原则如下：

- 首屏要先让用户感知到“这是一个高级桌面应用”，其次才是“这是一个网页聊天界面”。
- 中央主区必须持续服务于输入与阅读，不被导航和状态层抢走焦点。
- 所有玻璃感、模糊、阴影和透明度都应服务于层次，而不是成为视觉噱头。
- 聊天内容必须比装饰层更稳定、更可信、更易读。

## 4. 整体信息架构

主界面采用三段式结构，但视觉语言与层级关系重做：

1. 左侧 Sidebar：固定导航 + 会话列表
2. 顶部 Toolbar：轻工具栏
3. 中央 Chat Workspace：聊天主舞台 + 固定底部输入区

不引入常驻右侧信息栏。与模型、状态、导出相关的附加信息应以内嵌入口、弹层或轻量菜单承载，而不是挤压中央聊天区。

## 5. 视觉系统

### 5.1 气质关键词

- 深色
- 玻璃感
- 原生桌面
- 安静
- 克制
- 沉浸

### 5.2 色彩方向

整体从暖白 Atelier 体系切换到深石墨与烟灰层叠体系：

- App 外层背景：深石墨、炭灰、冷棕灰渐变
- Sidebar：更深一层的半透明深灰玻璃
- Chat Workspace：比 Sidebar 略亮的稳定深色内容面板
- 文本：避免纯白，使用低刺激的高亮灰白
- 强调色：保留少量系统级或品牌级高亮，仅用于发送按钮、激活态与关键操作

使用原则：

- 不使用纯黑铺底，避免压抑与廉价夜间模式感。
- 不使用高饱和大面积主色块。
- 通过透明度、模糊、亮暗差和留白建立层级，而非依赖重边框。

### 5.3 材质与边界

核心材质语法：

- 深色半透明底板
- 中高强度 `backdrop-blur`
- 极轻内高光
- 低对比分割线
- 柔和大半径圆角

禁止项：

- 厚重描边
- 强玻璃高光造成的廉价拟态
- 过度发蓝的赛博风透明效果
- 大面积纯色卡片堆叠

### 5.4 排版

排版上不沿用参考图中的媒体内容首页语义，而保留产品型界面的清晰和效率：

- 品牌、区块标题、少量导航标题可以更有桌面应用感
- 会话标题、消息正文和输入文字优先可读性
- 小字与 meta 信息不应过浅，必须保证深色背景下的稳定对比

## 6. Sidebar 设计

### 6.1 结构

Sidebar 采用上下两段结构：

- 上半区：固定导航区
- 下半区：会话列表区

固定导航区包含：

- 极简品牌区，显示 `Echo`
- 搜索入口
- 3 到 5 个固定导航项，例如 `Chat`、`Agent`、`Skills`、`Settings`

会话列表区包含：

- `Recent Chats` 或同语义标题
- 按时间或最近活跃逻辑组织的会话列表
- 当前激活会话高亮
- 重命名、删除、置顶等操作默认隐藏，悬停再显示

### 6.2 视觉表现

Sidebar 需要明确像原生桌面侧栏，而不是网页后台菜单：

- 背景比中央区更暗
- 具备磨砂半透明与模糊层
- 搜索框采用内嵌式半透明胶囊
- 图标默认压低亮度
- 激活态通过轻玻璃底、文字加重和小范围强调色实现

### 6.3 交互原则

- 导航区不随会话列表滚动
- 会话列表滚动时仍保持上方导航稳定
- hover 必须清晰，但不能出现高亮闪烁式反馈
- 上下区块之间使用极轻分隔，不做重卡片切断

## 7. 顶部 Toolbar 设计

顶部栏定位为轻工具栏，而不是状态中心。

建议结构：

- 左侧：品牌或当前区域名
- 中部：留白
- 右侧：少量关键操作，例如搜索、模型入口、导出、更多菜单

设计原则：

- 和 Sidebar 属于同一套玻璃材质，但更轻、更薄、更通透
- 不承担复杂信息密度
- 不与中央聊天内容争抢焦点
- 通过极细分隔线与微妙高光建立“漂浮在内容上方”的桌面感

## 8. 中央 Chat Workspace 设计

中央区保持 Echo 的聊天工作台定位，不转向内容首页。

建议拆成三层：

1. 内容底板
2. 消息层
3. 固定底部输入层

内容底板应为稳定的深色内容面板，用于承接长时间阅读。它可以保留轻微玻璃质感，但应弱于 Sidebar，避免聊天内容漂浮不稳。

主区原则：

- 不引入大推荐卡片
- 不做媒体型栏目分区
- 有消息时始终优先服务会话阅读
- 空状态也应以邀请发起对话为主，而不是展示大量内容模块

## 9. 消息区重绘策略

本次采用中度重绘，而非彻底推翻聊天模型。

### 9.1 用户消息

- 保持右侧对齐
- 使用克制的深灰半透明胶囊气泡
- 弱化传统即时通讯配色感
- 视觉上更像输入痕迹，而不是强品牌动作块

### 9.2 助手消息

- 弱化“气泡”概念，更接近内容回答卡
- 主回答面板比页面略亮，但仍在深色体系内
- 通过轻底色差、大圆角和少量阴影建立可读性
- 正文必须始终拥有最高阅读优先级

### 9.3 辅助内容

代码块、工具调用、引用、附件、推理状态等辅助模块应以内嵌次级卡片呈现：

- 明显低于主回答权重
- 支持折叠或降噪处理
- 避免把回答割裂成过多技术碎片

## 10. Composer 输入区设计

输入区是整个主界面最稳定的交互锚点。

要求如下：

- 固定在底部
- 和中央阅读列对齐
- 使用深色玻璃托盘式容器
- 聚焦时轻微抬升，而不是强高亮外框

内部能力继续复用现有结构：

- 输入文本
- 模型选择
- 附件
- Skills
- MCP
- 发送与停止生成

视觉目标：

- 比普通消息更稳定、更可信
- 比主回答更轻，不喧宾夺主
- 成为用户在任何状态下都能立即理解的主操作入口

## 11. 与现有代码结构的对应关系

本次设计应建立在已有边界上渐进落地：

- `src/features/app/AppView.tsx`
  - 负责整体壳层、顶栏、布局节奏
- `src/components/Sidebar.tsx`
  - 承接新侧栏结构与玻璃材质
- `src/components/ChatView.tsx`
  - 作为中央工作台入口，承接新的阅读舞台样式
- `src/components/chat/*`
  - 调整消息框、回答卡、辅助块的视觉层级
- `src/components/Composer.tsx`
  - 重绘底部输入托盘与聚焦反馈
- `src/index.css`
  - 重建全局深色玻璃壳层样式
- `src/styles/design-tokens.css`
  - 补充或替换设计 token，支持新配色与材质变量

实施原则：

- 先改壳层，再改侧栏，再改聊天区和输入区
- 优先复用现有逻辑与状态控制
- 尽量避免对测试语义和核心交互造成结构性破坏

## 12. 桌面端壳层约束

由于当前应用运行在 Electron 桌面环境中，视觉设计需要服从以下实现边界：

- 顶部需要兼容现有窗口拖拽区域，不能让工具栏按钮侵入不可点击的拖拽层
- macOS 下需要为 traffic lights 与浮动侧栏开关预留空间
- 紧凑宽度下 Sidebar 仍需支持收起和浮动展开，不能因为玻璃样式导致可用性下降
- `chat`、`agent`、`settings` 三种主视图共享同一外层壳体，因此壳层语言应统一，聊天页作为首个重点精修对象

设计含义如下：

- 顶部 Toolbar 的视觉可以更轻，但点击热区必须清晰
- 玻璃层与阴影层不能影响拖放附件、Toast、错误条和浮动按钮的可见性
- 收起侧栏后的主内容区应保持完整节奏，不能留下突兀的空白断层

## 13. 状态与异常处理原则

新视觉不应遮蔽现有关键状态反馈，尤其是聊天工作流中的异常与临时状态。

必须保留并重新适配的状态包括：

- 加载态
- 会话删除后的撤销提示
- 错误条
- 文件拖拽覆盖态
- 生成中状态
- 空状态与首条会话状态切换

原则如下：

- 关键状态优先保证可见性，再考虑玻璃质感统一
- 错误、警告、撤销信息应继续拥有明确层级，不可因为深色弱对比而隐身
- 覆盖态和 Toast 应与新壳层协调，但不可以牺牲可发现性

## 14. 分阶段实施建议

### 阶段 1：壳层和设计 token 切换

- 建立深色玻璃背景、主面板、侧栏材质变量
- 重做 app shell 外层层次
- 让整体界面先脱离 Atelier 暖纸面风格

### 阶段 2：Sidebar 与顶部 Toolbar

- 重构为上方固定导航 + 下方会话列表
- 加入搜索入口与安静的激活态
- 建立顶部轻工具栏

### 阶段 3：聊天内容区与输入区

- 调整消息卡片层级
- 重绘助手回答容器与用户消息气泡
- 改造底部输入托盘

### 阶段 4：细节抛光

- 优化 hover、focus、分隔线与阴影
- 校准深色背景下文本对比度
- 补齐不同窗口宽度下的响应表现

## 15. 测试与验证要求

实现阶段需要围绕以下方向补齐验证：

- `Sidebar` 导航、会话列表、悬停操作在新结构下的渲染与交互测试
- `AppView` 中侧栏开合、空状态、聊天状态、拖拽状态的布局稳定性测试
- 消息容器与输入区在深色主题下的类名和状态切换测试
- 至少一次桌面端手动验收，确认拖拽区、窗口按钮避让、滚动区域和浮动开关表现正常

## 16. 验收标准

满足以下条件则视为方向落地成功：

- 第一眼感受接近原生深色桌面应用，而非网页聊天页
- 左侧栏明确具备磨砂半透明玻璃质感
- 顶栏、侧栏、中央区形成稳定的深色层级关系
- 中央区仍明显是 Echo 的聊天工作台，而不是内容首页
- 输入区始终是稳定主锚点
- 消息区在深色体系下依然保持良好可读性

## 17. 风险与控制

主要风险：

- 过度模仿参考图，导致 Echo 看起来像媒体播放器
- 玻璃效果过强，削弱文本阅读稳定性
- 深色体系处理不当，造成对比不足和可访问性问题
- 在现有组件上重绘时引发不必要的结构性改动

控制策略：

- 始终以聊天工作流为主轴校验设计
- 将玻璃质感限制在壳层与容器，而不是覆盖所有文本载体
- 先做壳层与层次，再逐步提高细节强度
- 通过分阶段实施降低回归风险
