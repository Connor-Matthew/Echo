# 本地用户画像系统设计

日期：2026-03-17  
状态：待实现

## 1. 背景

当前项目里与“记忆”相关的能力有两类：

- `soul` 链路中的本地 `memory.md / journal / SOUL.md`
- 可选开启的 MemOS Cloud 跨会话记忆

但这两类能力都还没有真正解决“AI 如何稳定理解这个用户是谁”这个问题。

如果只靠 `SOUL`，用户信息容易和 AI 自身人格混在一起；如果只靠外部记忆检索，信息又容易碎片化、短期化，难以形成稳定的人物画像。用户现在真正需要的是一套独立于 `soul` 的本地画像系统，让 Echo 能在长期相处中逐步形成对用户的保守、可纠偏、可审阅的理解。

本次设计的目标，不是做一个“什么都记”的大仓库，而是做一个完全本地、自动沉淀、结构清晰的人物画像系统，用于支持：

1. 识别用户的偏好与习惯
2. 识别用户的人生背景与长期状态
3. 识别用户与 Echo 的关系模式、边界与信任方式

同时，这套系统必须满足几个前提：

- 完全本地，不同步云端
- 用户无须单独安装数据库
- 自动更新，但始终可手动查看、编辑、停用、删除
- 中等强度影响聊天，不允许 AI 过度自信地“演懂你”
- 重要画像项尽量保留依据，便于日后纠偏

## 2. 核心目标

本方案要解决的，不是“每天生成一份流水账”，而是让 Echo 获得一套逐渐稳定、又不会失控漂移的用户画像机制。

第一版的核心目标如下：

1. 新增独立于 `soul` 的“用户画像”能力域，避免与 AI 自身人格互相污染。
2. 使用 Echo 自带的本地 SQLite 作为唯一真源，用户无需额外安装数据库。
3. 每天自动生成一份“用户日摘要”，记录用户当天做了什么、关注什么、呈现出什么状态。
4. 基于最近一段时间的日摘要，自动更新三层结构化画像。
5. 对重要画像项保留证据链，对轻量偏好允许较弱证据。
6. 在设置中提供独立入口，允许浏览、编辑、停用、删除画像项。
7. 聊天时只注入压缩后的当前画像上下文，不直接注入数据库原文或完整日记。

本方案明确不在第一版实现以下内容：

- 云同步
- 复杂全文搜索
- 批量回滚
- 历史旧会话全量回灌
- 多端画像合并

## 3. 总体设计

本地用户画像系统采用：

- `SQLite` 作为结构化真源
- `Markdown snapshot` 作为可读导出视图

换句话说，真正可写、可查询、可停用的画像数据全部存在数据库里；用户在设置页里看到的“当前人物画像”则由数据库整理生成一份 Markdown 快照。

这样设计的原因：

1. 画像是结构化数据，不只是长文档。
2. 后续会有“编辑某一条、删除某一条、查看某条依据、停用某一层”这类操作，数据库更适合。
3. 同时保留 Markdown 快照，便于用户快速理解当前画像，也便于调试与导出。

因此第一版的数据流为：

`用户消息 -> 日摘要 -> 画像更新 -> Markdown snapshot -> 聊天注入`

## 4. 分层模型

人物画像拆成三个稳定层：

### 4.1 偏好与习惯

这一层记录用户在长期互动中表现出的偏好、节奏和习惯，例如：

- 更偏好简洁还是细致的表达
- 做决策时偏理性分析还是感受驱动
- 工作与生活节奏是否呈现明显模式
- 与 AI 协作时更喜欢直接执行还是先讨论

这一层允许相对更快地自动更新，因为它主要影响“如何回应用户”，风险相对较低。

### 4.2 人生背景与长期状态

这一层记录用户较稳定、长期、对理解语境有帮助的背景信息，例如：

- 长期项目与职责
- 长期生活结构或关系角色
- 反复出现的重要现实处境
- 需要被持续尊重的敏感背景

这一层必须更保守。单日出现的信息不应直接进入稳定画像，必须在多次明确提及时才可沉淀。

### 4.3 关系画像

这一层记录用户与 Echo 的互动方式，不写进 `SOUL`，只服务于“Echo 如何更准确地理解和回应这个用户”。

可记录的信息包括：

- 用户希望 Echo 采取怎样的语气与协作姿态
- 用户对边界、直率、安抚、节奏的偏好
- 关系中的信任方式、纠错方式、默认合作模式

这层的目标不是制造感情设定，而是帮助 Echo 形成更稳的互动方式。

## 5. 数据模型

第一版建议至少包含以下表：

### 5.1 `daily_user_notes`

按天存用户日摘要，是整套画像系统的原材料层。

建议字段：

- `id`
- `date`：`YYYY-MM-DD`
- `summary_markdown`
- `source_message_count`
- `created_at`
- `updated_at`

职责：

- 记录当天用户做了什么
- 记录当天反复出现的话题、状态、关注点
- 提供后续画像更新的依据

注意：它不是最终画像，不直接下稳定人格判断。

### 5.2 `profile_items`

结构化画像本体，每条代表一个稳定或半稳定判断。

建议字段：

- `id`
- `layer`：`preferences` / `background` / `relationship`
- `title`
- `description`
- `confidence`
- `status`：`active` / `disabled`
- `evidence_strength`
- `last_confirmed_at`
- `created_at`
- `updated_at`

职责：

- 存放当前有效画像项
- 支持手动编辑、停用、删除
- 支持后续聊天注入时筛选

### 5.3 `profile_evidence`

用于把重要画像项和其依据关联起来。

建议字段：

- `id`
- `profile_item_id`
- `daily_note_id`
- `excerpt`
- `weight`
- `created_at`

职责：

- 让重要画像项尽量“可追溯”
- 支持设置页里“查看依据”
- 支持后续自动提权或降权

### 5.4 `profile_snapshots`

用于保存可读快照，便于调试、回看与导出。

建议字段：

- `id`
- `markdown`
- `created_at`

职责：

- 保留某一时刻的完整画像视图
- 为设置页和导出提供稳定来源

## 6. 存储位置与本地数据库

本方案使用 Echo 内置 SQLite，不要求用户安装外部数据库。

建议存储位置放在 Electron 本地数据目录下，与其他本地状态并列，例如：

```text
app.getPath("userData")/profile/
├── profile.db
└── snapshots/
    └── current-profile.md
```

其中：

- `profile.db` 是唯一真源
- `current-profile.md` 是当前画像快照导出

这样做的好处是：

- 完全本地
- 应用可自动建库建表
- 不受用户家目录已有文件结构影响
- 与现有 `~/.echo/memory/` 的 `soul` 数据域保持边界

## 7. 每日沉淀流程

### 7.1 每日用户摘要生成

系统每天自动基于“当天所有用户消息”生成一条 `daily_user_notes`。

摘要目标：

- 简洁记录用户今天做了什么
- 记录主要关注点、推进事项、状态变化
- 可适度概括，但不直接写成稳定画像

风格要求：

- 客观、简洁、中文
- 偏事实与观察
- 不写 AI 自述
- 不用诗化表达

如果当天没有有效用户消息，则不生成日摘要。

### 7.2 画像更新

画像更新器周期性读取最近一段时间的 `daily_user_notes`，尝试把重复出现的模式提升为 `profile_items`。

推荐规则：

- 1 次出现：只保留在日摘要，不进入稳定画像
- 2 次出现：可作为弱信号候选
- 3 次及以上：可升级为较稳定画像项

但对不同层设置不同门槛：

- `偏好与习惯`：允许较快升级
- `人生背景与长期状态`：必须更谨慎
- `关系画像`：只记录互动模式，不做浪漫化解释

### 7.3 快照生成

每次画像更新成功后，重新生成 Markdown 快照。

快照结构建议为：

```md
# 用户画像

## 偏好与习惯
- ...

## 人生背景与长期状态
- ...

## 关系画像
- ...

## 最近依据
- 2026-03-17: ...
```

该快照只用于阅读和导出，不作为主存储。

## 8. 聊天注入策略

聊天时不直接把数据库原文、所有日摘要、所有证据一并注入。

第一版只注入一份压缩后的“当前画像上下文”，由激活中的 `profile_items` 生成。

注入原则：

1. 只注入 `active` 状态的画像项。
2. 按层精选，不追求全量。
3. 重要背景和关系边界优先，低价值细节不注入。
4. 以“中等影响”为原则，只改变理解和回应方式，不允许 AI 假装完全知道用户全部人生。

建议输出结构：

```text
<user_profile>
## 偏好与习惯
- ...

## 人生背景与长期状态
- ...

## 关系画像
- ...
</user_profile>
```

该注入块应与 `SOUL.md` 注入严格分离，避免用户画像污染 AI 自身人格。

## 9. 设置页设计

新增独立设置项：`用户画像`。

页面建议分为四个入口：

1. `偏好与习惯`
2. `人生背景与长期状态`
3. `关系画像`
4. `每日记录`

### 9.1 画像层视图

每条画像项以卡片展示，包含：

- 标题
- 描述
- 置信度或强度
- 最近更新时间
- 是否启用
- 查看依据

支持操作：

- 编辑
- 删除
- 停用 / 启用
- 手动新增

### 9.2 每日记录视图

按日期列表展示 `daily_user_notes`，类似现有 `journal` 浏览方式，但内容是用户日摘要而非 AI 日记。

第一版不要求支持复杂筛选，只需：

- 左侧日期列表
- 右侧 Markdown 渲染
- 默认选中最近一条

### 9.3 快照视图

设置页可额外提供“当前画像快照”只读区域，帮助用户整体审阅 Echo 目前如何理解自己。

## 10. 风险与边界

### 10.1 过度自信

风险：模型基于少量材料形成过强判断。

控制方式：

- 采用分层与重复证据门槛
- 只做中等影响注入
- 重要画像尽量保留依据

### 10.2 画像漂移

风险：短期情绪、偶发事件被写成长期判断。

控制方式：

- 日摘要与稳定画像分层
- 画像更新按重复出现提权
- 允许手动停用和删除

### 10.3 用户失去控制权

风险：系统自动更新后，用户无法理解或纠正。

控制方式：

- 设置页必须可审阅
- 画像项必须可编辑、停用、删除
- 停用项不得再注入聊天

### 10.4 与 `soul` 混淆

风险：用户背景被写进 AI 自身人格，或 `SOUL` 反向影响用户画像。

控制方式：

- 新系统完全独立于 `soul`
- 注入块分离
- 数据域分离
